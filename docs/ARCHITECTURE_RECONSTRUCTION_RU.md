# Реконструкция архитектуры VK Publisher

Документ фиксирует, как привести репозиторий в понятный и поддерживаемый вид.

Дата: 2026-03-29

---

## 1. Проблема в текущем состоянии

Сейчас в репозитории смешаны:
- Python runtime (pipeline, fetch/process/moderate/publish, web UI),
- PHP SDK/библиотека для VK,
- частично пересекающиеся документы.

Это приводит к:
- высокой когнитивной нагрузке,
- конфликтующим импортам и «двум истинам» в коде,
- сложному онбордингу и нестабильному CI.

---

## 2. Целевое состояние

### Вариант А (рекомендуемый)
Разделить на 2 репозитория:
1. `vk_publisher_app` (Python приложение)
2. `vk_publisher_php` (PHP библиотека)

### Вариант B (если пока нельзя разделить)
Оставить один репозиторий, но физически развести:

```text
apps/python/
packages/php/
docs/
```

И ввести единые правила для imports, CI, docs.

---

## 3. Поэтапный план (без остановки разработки)

### Этап 1 — Инвентаризация (1–2 дня)
- Зафиксировать список «боевых» entrypoint-файлов.
- Пометить legacy-модули (deprecated).
- Зафиксировать владельца каждого модуля.

Артефакты:
- `docs/CODE_OWNERSHIP.md`
- `docs/ENTRYPOINTS.md`

### Этап 2 — Нормализация структуры (2–4 дня)
- Ввести базовую иерархию папок для Python и PHP.
- Убрать дублирующие адаптеры/клиенты (или пометить переходный слой).
- Переписать импорты на единый стиль.

Критерий готовности:
- нет «двойных» реализаций без причины,
- все импорты deterministic из корня проекта.

### Этап 3 — Тесты и CI (2–3 дня)
- Разделить пайплайны CI для Python и PHP.
- Добавить обязательные минимальные проверки:
  - Python: `pytest -q`
  - PHP: `vendor/bin/phpunit --testdox`, `vendor/bin/phpstan analyse --no-progress`
- Добавить smoke-проверку запуска приложения.

### Этап 4 — Документация (1–2 дня)
- Поддерживать только один индекс docs: `docs/README_RU.md`.
- Удалить/объединить дублирующиеся инструкции.
- Добавить «START HERE» для нового пользователя.

## 4. Быстрые победы (можно делать сразу)

1. Везде явно разделить термины:
   - **App (Python)**
   - **Library (PHP)**
2. Любой новый модуль создавать только в целевой папке (не в legacy).
3. Для каждого PR: короткий changelog + обновление документации.

### Уже сделано в рамках PR #1 (cleanup)

- Перенесены legacy-модули:
  - `src/vk_api_client.py` -> `src/legacy/vk_api_client.py`
  - `src/publisher/vk_publisher.py` -> `src/legacy/vk_publisher.py`
- На старых путях оставлены compatibility-shim модули с `DeprecationWarning`.
- Зафиксированы канонические точки:
  - VK client (async): `src.infrastructure.vk_api_client.VKClient`
  - Pipeline runtime: `src.workers.pipeline.PipelineWorker`
