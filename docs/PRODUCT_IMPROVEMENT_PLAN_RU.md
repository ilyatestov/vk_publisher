# План улучшений VK Publisher (GUI, дедупликация, VPS/Windows, защита, релиз)

Документ описывает **практический roadmap**: что и в каком порядке внедрять, чтобы получить удобный сервис автопостинга с веб-настройкой, защитой, антидублями и мультирепостом в соцсети.

> Навигатор по документации: [`docs/README_RU.md`](./README_RU.md).

---

## 1) Целевой результат

После реализации этапов ниже проект должен уметь:

1. Работать как веб-сервис на VPS и локально на Windows.
2. Управляться из браузера (телефон/ПК) через полноценный GUI с разделением по функциям.
3. Автоматически собирать контент, проверять дубли (точные и «похожие»), публиковать по расписанию.
4. Добавлять в конце поста блок «Подписывайтесь на мои соцсети» с централизованной настройкой.
5. Иметь базовую защиту от DDoS/брутфорса и от блокировок API (лимиты, ретраи, очереди).
6. Иметь «инструкцию для новичков» + отдельные сценарии установки: VPS и Windows.

## 3) Оптимизация БД и уникальности контента

### 3.1 Что хранить

Рекомендуется расширить БД (SQLite/PostgreSQL) следующими сущностями:

- `sources` — источники (тип, URL, активность, период опроса).
- `source_items` — сырые элементы из парсеров (`external_id`, `url`, `published_at`, `raw_text`, `raw_hash`).
- `post_fingerprints` — нормализованные отпечатки текста:
  - `exact_hash` (SHA256 нормализованного текста),
  - `simhash` или `minhash` для похожих текстов,
  - `language`, `token_count`.
- `publish_jobs` — задания публикации (status, scheduled_at, attempts, error_message).
- `destinations` — куда дублировать финальный пост (Telegram channel, webhook, etc.).

### 3.2 Алгоритм антидублей

1. Нормализация текста: lowercase, удаление ссылок/emoji/служебных хвостов, лемматизация (опц.).
2. Exact check по `exact_hash` (быстро, индекс UNIQUE).
3. Near-duplicate check:
   - либо cosine similarity по эмбеддингам,
   - либо simhash/hamming distance (дешевле для VPS).
4. Если найден дубль: не публиковать, записать в логи причину.
5. Если текст «похож, но не дубль»: можно отправить в модерацию с пометкой risk-score.

### 3.3 Индексы и производительность

- Индексировать: `source_items(external_id, source_id)`, `publish_jobs(status, scheduled_at)`, `post_fingerprints(exact_hash)`.
- Для PostgreSQL: добавить `pg_trgm` для поиска похожих текстов.
- Вводить TTL/архивацию: старые сырые записи (90–180 дней) переносить в cold storage.

## 4) Улучшение GUI (веб-панель)

Сделать GUI модульным (по вкладкам), каждая функция настраивается отдельно:

1. **Dashboard**: состояние, ошибки, очередь публикаций, последние посты.
2. **Источники**: добавить/изменить/отключить RSS, сайты, VK-паблики.
3. **Обработка текста**: рерайт, длина, тон, хештеги, фильтры стоп-слов.
4. **Антидубли**: окно проверки, пороги схожести, white/black list.
5. **Планировщик**: расписание, дневные лимиты, «не публиковать ночью».
6. **Каналы публикации**: VK + пересылка в Telegram/webhook/другие соцсети.
7. **Footer/Соцсети**: шаблон конца поста и список ваших ссылок.
8. **Безопасность**: логины, токены, список IP, журнал входов.

Минимальный UX-уровень:

- Валидация полей прямо в форме.
- Кнопка «Проверить подключение» для каждого интеграционного токена.
- Кнопка «Тестовый пост» с dry-run.
- Профили настроек: «Новостной», «Маркетинг», «Осторожный (жесткая дедупликация)».

## 5) Автопубликация и сверка дублей

Рекомендуемый pipeline:

1. Fetch из активных источников.
2. Нормализация и fingerprints.
3. Dedup stage (exact + near).
4. Rewrite/Moderation (если включено).
5. Scheduler ставит в очередь.
6. Публикация в VK.
7. Пост-публикационный fan-out (Telegram/webhooks/другие каналы).
8. Логирование и метрики.

Технически важно:

- Идемпотентность: один `job_id` не публикуется повторно.
- Retry-policy: экспоненциальная пауза + ограничение попыток.
- Dead-letter queue для «плохих» задач.

## 6) Защита от DDoS, блокировок и злоупотреблений

### На уровне веб-доступа

- Nginx reverse proxy + `limit_req` / `limit_conn`.
- fail2ban по логам Nginx/Uvicorn.
- Обязательная авторизация в GUI.
- CORS только для доверенных origin.

### На уровне API интеграций

- Троттлинг вызовов VK API (глобальный и по методу).
- Circuit breaker при массовых ошибках API.
- Backoff + jitter при 429/5xx.
- Поддержка прокси/ротации endpoint (опционально, если легально и по правилам API).

### На уровне данных

- Шифрование секретов (минимум `.env`, лучше vault/secret manager).
- Регулярный backup БД + проверка восстановления.

## 7) Установка и эксплуатация: VPS vs Windows

### Вариант 1: VPS (рекомендуется для 24/7)

- docker-compose профиль `prod-vps`.
- Компоненты: app + db + redis + nginx + monitoring.
- HTTPS (Let's Encrypt), домен, системный автозапуск.
- Доступ к GUI: `https://your-domain` с авторизацией.

### Вариант 2: Windows (локальный запуск)

- Скрипт-установщик (`.bat`/PowerShell), который:
  - создаёт venv,
  - ставит зависимости,
  - копирует `.env.example` в `.env`,
  - запускает web UI и API.
- В docs отдельный раздел «как открыть доступ из домашней сети» (опционально, с рисками).

Кратко по модулям, куда вносить изменения в первую очередь:

## 11) Минимальный MVP, который даст максимум пользы

Если делать только 5 вещей, то:

1. Реальные CRUD-эндпоинты для источников + GUI для них.
2. Exact + near dedup.
3. Планировщик + идемпотентная очередь публикации.
4. Footer «мои соцсети» и fan-out после постинга.
5. Nginx rate-limit + авторизация GUI + подробный onboarding в docs.

Это уже даст заметный прирост стабильности, удобства и масштабируемости проекта.
