# Технический аудит проекта VK Publisher

Дата аудита: **2026-03-29**

## 1) Оптимизация производительности

### Что найдено

1. В `pipeline` используются последовательные шаги с ожиданием решения модерации на каждом посте, что снижает throughput при пиках нагрузки.
2. В `web_parser` создаётся новый `aiohttp.ClientSession` на каждый запрос, а не переиспользуется общий пул соединений.
3. В fetch-циклах есть фиксированные sleep-интервалы без адаптации к latency и backlog очередей.

### Рекомендации

- Ввести единый `aiohttp.ClientSession` (lifecycle на уровне приложения).
- Добавить backpressure-метрики и динамический poll interval.
- Вынести обработку модерации в отдельные конкурентные задачи (bounded semaphore).
- Добавить профилирование времени этапов (`fetch`, `process`, `moderate`, `publish`) и 95p latency.

## 3) Поиск уязвимостей

### Потенциальные риски

1. **SSRF / небезопасные исходящие запросы**
   - Сборщики контента принимают URL-источники; нужна allowlist-политика и блок private CIDR.
2. **Утечки секретов**
   - Токены и ключи должны храниться только в env/secret manager, не в `config/*.json`.
3. **Чувствительные данные в логах**
   - Проверить, что токены не попадают в исключения/traceback.
4. **Слабый контроль зависимостей**
   - Нужны регулярные `pip-audit` и обновления lock-файла.

### Обязательные проверки в CI

```bash
pytest -q
bandit -r src
pip-audit
```
